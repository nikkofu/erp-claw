# 全局阶段路线总览

更新时间：2026-03-26

## 1. 文档目的

本文用于把当前 ERP Claw 的总体阶段路线整理成一个全局视图，帮助后续在设计、排期和实施上保持统一判断。

它和分阶段文档的关系是：

- 本文回答“整体怎么分阶段推进”
- `phase-1-coverage-status.md` 回答“当前平台底座做到哪里了”
- `phase-2-coverage-status.md`、`phase-3-coverage-status.md`、`phase-4-coverage-status.md` 回答“后续阶段分别要覆盖什么、优先级如何、当前准备度如何”

## 2. 当前全局判断

当前项目处于：

- `Phase 0/1 平台底座已完成第一批可运行基线`
- `Phase 1 控制面已经落地第一批可执行切片，但离完整交付仍有明显缺口`
- `Phase 2~4 仍处于设计完成、执行未开始状态`

因此，后续路线不应该继续扩展更多无业务承载的技术骨架，而应该沿着既定路线逐步完成：

1. 控制面实体化
2. 供应链闭环落地
3. AI 执行面强化
4. 企业级扩展

## 3. 阶段总表

| 阶段 | 名称 | 核心目标 | 主要产出 | 当前状态 |
| --- | --- | --- | --- | --- |
| Phase 0/1 | 平台底座与控制面骨架 | 跑通运行时、配置、租户上下文、策略审计、事件总线、agent gateway seam | 五个 runtime、基础中间件、迁移、event bus、command pipeline、smoke 流程，以及第一批控制面可执行切片 | 已完成第一批可执行切片 |
| Phase 2 | 供应链交易闭环 | 把采购、库存、审批、主数据等最小业务闭环落地 | 领域模型、命令处理、管理 API、读模型、闭环交易流 | 未开始 |
| Phase 3 | AI-Native Execution Strengthening | 把 Agent Workspace、任务状态机、工具运行时、知识检索、审批暂停/恢复做成真正的执行系统 | workspace 流式协议、agent session/task runtime、workflow orchestration、human handoff | 未开始 |
| Phase 4 | Enterprise Expansion | 把平台从“可用”扩展到“企业可规模化运行” | 开放集成平台、专属租户单元、增强搜索分析、深度治理 | 未开始 |

## 4. 阶段依赖关系

### 4.1 主依赖链

```text
Phase 0/1 平台底座
        |
        v
Phase 2 供应链交易闭环
        |
        +------------------+
        |                  |
        v                  v
Phase 3 AI执行强化    Phase 4 企业级扩展
```

### 4.2 关键依赖说明

- Phase 2 必须建立在已有平台底座之上，否则业务领域会被迫与底层技术装配耦合。
- Phase 3 必须建立在 Phase 2 的真实业务命令和审批对象之上，否则 Agent 只能操作空壳骨架。
- Phase 4 同时依赖 Phase 2 的业务厚度和 Phase 3 的执行治理成熟度，否则无法进入真正的企业扩展阶段。

## 5. 每个阶段的入口条件

### 5.1 Phase 2 入口条件

- 运行时、配置、迁移、事件总线、命令管道稳定可用
- 租户上下文和审计链路已经存在
- 本地开发和 smoke 验证流程可重复执行

当前判断：已基本满足。

### 5.2 Phase 3 入口条件

- 至少一条真实供应链业务闭环已经落地
- 审批、库存、采购、主数据等核心对象已有真实命令与状态
- Agent 运行的对象不再只是占位表和骨架接口

当前判断：尚不满足。

### 5.3 Phase 4 入口条件

- 多个核心业务域已进入稳定运行状态
- Agent 执行和人工接管已具备平台化治理能力
- 集成、检索、分析和租户隔离需求开始成为规模化瓶颈

当前判断：尚不满足。

## 6. 分阶段落地建议

### 6.1 推荐顺序

推荐严格按下面顺序推进：

1. `补齐 Phase 1 控制面主线`
2. `进入 Phase 2 供应链闭环`
3. `进入 Phase 3 AI 执行强化`
4. `进入 Phase 4 企业级扩展`

### 6.2 不建议的顺序

不建议在 Phase 2 之前提前大量投入以下方向：

- 高级向量检索
- 复杂多代理编排
- 插件市场化能力
- 多 cell 大规模拓扑
- 高级分析大屏

原因很直接：如果真实业务命令、审批流和库存/采购事务还没有落地，这些能力只会围绕空壳结构旋转。

## 7. 当前建议的文档组合

为了支撑“全局思考、分阶段落地”，建议把路线文档固定为下面 5 份：

- `docs/phase-roadmap-overview.md`
- `docs/phase-1-coverage-status.md`
- `docs/phase-2-coverage-status.md`
- `docs/phase-3-coverage-status.md`
- `docs/phase-4-coverage-status.md`

这样做的价值是：

- 总览文档负责阶段关系和决策顺序
- 分阶段文档负责范围、功能、优先级、依赖和状态
- 每个阶段未来都可以继续演化成 backlog、里程碑或实施计划

## 8. 下一步建议

从实施角度看，最合理的下一步不是直接开始 Phase 3 或 Phase 4，而是：

1. 先把 Phase 1 控制面主线补齐
2. 立刻进入 Phase 2 最小供应链闭环

只有这样，后续 AI 执行面才有真实业务对象可操作，Phase 4 的企业扩展也才有清晰的约束对象。

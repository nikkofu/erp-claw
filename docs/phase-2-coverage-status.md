# Phase 2 覆盖领域、模块、功能清单、优先级与实现情况

更新时间：2026-03-25

## 1. Phase 2 定位

Phase 2 的名称是：

- `Supply-Chain Transaction Loop`
- 中文可表述为：`供应链交易闭环`

它是整个项目第一个真正落地业务价值的阶段，目标不是“做一个大而全 ERP”，而是做成一条从主数据到采购、库存、审批，再到应付/应收基础的最小可用闭环。

## 2. Phase 2 目标

### 2.1 核心目标

- 建立可被真实使用的供应链最小业务链路
- 让业务命令、审批和库存变化进入真正的事务域
- 为 Phase 3 的 Agent 执行面提供真实可调用的业务对象

### 2.2 推荐的最小闭环

建议 Phase 2 的最小闭环按下面口径定义：

1. 主数据建立
2. 发起采购需求或采购订单
3. 走审批
4. 收货入库
5. 形成库存可用状态
6. 形成应付基础记录

如果范围允许，再补一条轻量销售出库链作为第二条闭环。

| 有界上下文 | 功能范围 | 优先级 | 当前实现情况 | 说明 |
| --- | --- | --- | --- | --- |

| Procurement | 请购、采购订单、供应商事务状态、收货计划 | P0 | 未开始 | 这是供应链闭环的起点之一，应优先建立采购命令和状态机。 |

| Approval and Workflow | 审批定义、审批实例、人工任务、流程推进 | P0 | 未开始 | 采购与高风险交易需要审批，不应延后到 Phase 3。 |
| Receivable and Payable Basics | 应付单、应收单、付款计划、开票基础引用 | P1 | 未开始 | Phase 2 只需要基础账务引用，不需要完整总账。 |

## 5. Phase 2 优先级清单

### 5.1 P0：必须先做的主线

- Master Data 基础实体
- Procurement 基础交易流
- Approval 基础审批流
- Inventory 入库与库存可用状态
- Command Handlers 与 Admin API
- Outbox 可靠发布主路径

### 5.2 P1：在主线跑通后补齐

- Payable Basics
- Sales Minimal Loop
- Query Read Models
- Workspace 查询入口

### 5.3 P2：可以作为收尾或 Phase 3 的衔接项

- 集成同步入口
- Agent 对业务命令的受控调用
- 较完整的报表与监控投影

## 6. 当前准备度判断

Phase 2 虽然尚未开始，但前置平台条件已有相当部分准备：

- 已有命令管道骨架：可承接命令处理、策略校验、审计记录
- 已有事件总线骨架：可承接领域事件发布
- 已有 worker / scheduler：可承接 outbox 轮询与时间驱动任务
- 已有迁移基线：可开始扩展业务表
- 已有租户上下文和 HTTP 接口层：可开始建设管理 API

但同时仍有几个明显缺口：

- 没有真实 Tenant / IAM 数据模型
- 没有审批与工作流实现
- 没有 read model 和 repository 层
- 没有任何一个业务上下文的领域模型

结论是：

- `Phase 2 可开工`
- `但需要先在 Phase 2 开头补齐少量 Phase 1 控制面实体化工作`

## 7. 推荐的分波次落地方式

### Wave 1：主数据 + 采购 + 审批骨架

- 供应商
- 商品
- 仓库
- 采购订单
- 审批模板与审批实例
- 基础 Admin API

### Wave 2：收货入库 + 库存真相

- 收货单
- 入库事务
- 库存台账
- 可用库存/预留库存计算
- 采购完成态推进

### Wave 3：应付基础 + 轻量销售

- 应付基础记录
- 付款计划引用
- 可选的销售出库最小链路
- 读模型与列表页支持

## 8. Phase 2 完成判定建议

只有当下面几件事成立时，才能认为 Phase 2 进入完成态：

- 主数据、采购、审批、库存形成一条可测试闭环
- 至少一条真实采购到收货入库链路可跑通
- 库存状态来自真实领域事务，而非临时脚本或工作流假写
- 应付基础引用已经建立
- Admin API 可以操作这条闭环
- 集成测试可以覆盖关键命令、审批与库存变化

## 9. 与 Phase 3 的衔接关系

Phase 3 要强化 Agent 执行面，但它必须建立在 Phase 2 真实业务对象之上。

因此，Phase 2 的交付应该为 Phase 3 提供：

- 可查询的业务对象
- 可执行的命令处理器
- 可审计的审批节点
- 可观测的库存与采购状态
- 可被工具化暴露的查询/命令边界

如果做不到这一点，Phase 3 的 Agent 只能操作空壳流程。
